## Copilot Profiles

### Always-on

- Emits `.github/copilot-instructions.md`, `.github/agents/*.agent.md`, `.github/instructions/*.instructions.md`, reusable skills, and this runtime guide.
- Use this when KirOpen should shape the repo's default Copilot behavior and still have the specialist agents available.

### Lite

- Emits `.github/copilot-instructions.md` plus `.github/agents/*.agent.md`.
- The global instructions stay intentionally slim and route structured spec work into `spec-mode` to reduce context bloat.

### Agent-only

- Emits only `.github/agents/kiropen.agent.md` and `.github/agents/spec-mode.agent.md`.
- Use this when the team wants KirOpen only through explicit primary-agent selection, without repo-wide Copilot instructions.

## Known Issues

### Plan Mode and Todo Tool Compete With Spec Mode

Copilot's built-in plan mode and `#todo` tool can interfere with the spec workflow's own phase progression. If plan mode activates during a spec session, it may auto-advance phases or create parallel task tracking that conflicts with the spec's requirements → design → tasks gates.

Do this:
1. Do not use plan mode or `#todo` during spec workflow sessions.
2. After each phase, tell the agent what you want next — continue, revise, or generate remaining phases together.
3. If plan mode activates unexpectedly, tell the agent: `Do not use plan mode. Follow the spec workflow phase gates.`

### Lite Mode Intentionally Omits Full Repo Scaffolding

Lite mode does not emit path-scoped instructions, reusable skills, or runtime extras. That is intentional. If you later want repo-wide steering wrappers or reusable porting skills on disk, regenerate in Always-on mode instead of manually piecing those files together.

## Limits

- This guide documents behavior; it does not change runtime policy.
- Runtime-level instructions may supersede repository templates.
- Copilot behavior varies across surfaces (VS Code, CLI, github.com coding agent). Not all features are available everywhere.
